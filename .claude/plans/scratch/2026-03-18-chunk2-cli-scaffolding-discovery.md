## Metadata
- **Date:** 2026-03-18
- **Complexity:** large
- **Tech Stack:** TypeScript, Node.js, npm package, Commander.js, tsup (bundler)

## Objective
Build an npm CLI package (`syntaur`) that scaffolds the `~/.syntaur/` directory structure and generates properly-formatted markdown files for missions and assignments per the Syntaur protocol spec.

## User's Request
Create a detailed implementation plan for Chunk 2 (CLI Scaffolding Tool) of the Syntaur project. The CLI must support three commands:
- `syntaur init` — initialize `~/.syntaur/` directory with `config.md`
- `syntaur create-mission` — create a new mission folder with all required files
- `syntaur create-assignment` — create a new assignment folder inside a mission with all required files
- Handle one-off assignments (auto-wrap in a single-assignment mission)
- npm package skeleton (package.json, bin entry, TypeScript setup)

## Codebase Overview
This is a **greenfield project**. The `~/syntaur/` directory currently contains only:
- `.claude/plans/` — plan documents (high-level-plan.md, chunk1 spec)
- `docs/protocol/` — spec.md and file-formats.md (the completed Chunk 1 deliverables)
- `examples/sample-mission/` — a complete realistic example mission folder demonstrating all file formats

There is no existing code, no package.json, no tsconfig.json. Everything must be created from scratch.

## Patterns Discovered

| Pattern | Reference File | Description |
|---------|---------------|-------------|
| YAML frontmatter + markdown body | All protocol files | Every file uses `---` delimited YAML frontmatter for structured fields, markdown body for prose |
| Slug as folder name and frontmatter field | `examples/sample-mission/assignments/*/assignment.md` | The `slug` field in frontmatter always matches the containing folder name |
| UUID v4 for entity identity | `examples/sample-mission/mission.md` | Missions and assignments have UUID `id` fields; resources/memories do not |
| RFC 3339 timestamps everywhere | All files | Format: `2026-03-18T14:30:00Z` — always UTC |
| Absolute paths, never `~` | `docs/protocol/spec.md` section 8 | All filesystem paths in frontmatter must be expanded absolute paths |
| Underscore prefix for derived files | Directory structure | `_index-*.md`, `_status.md` — signals "do not edit manually" |
| Empty templates for optional files | `examples/sample-mission/assignments/write-auth-tests/` | Supporting files (scratchpad, handoff, decision-record) created as stubs with `Count: 0` |
| Commander.js for CLI | Web research | Industry-standard for Node.js CLI tools; full TypeScript support, subcommand routing |
| tsup for building | Web research | Modern TypeScript bundler; produces clean dist/ output for npm packages |

## One-Off Assignment Handling

The high-level plan states: "One-off assignments are single-assignment missions. Keeps the data model uniform. CLI tooling reduces the ceremony."

Implementation approach for `syntaur create-assignment` when no mission is specified (or a `--one-off` flag is used):
1. Generate a mission slug from the assignment title (e.g., assignment title "Fix login bug" becomes mission slug `fix-login-bug`)
2. Create the full mission folder structure (manifest, mission.md, agent.md, claude.md, empty indexes, resources/, memories/)
3. Create the single assignment inside it
4. The mission title matches the assignment title (since it's a 1:1 relationship)
5. User experiences this as a single command, not two

## CLI Command Signatures (Proposed)

```
syntaur init
  --force       # Overwrite existing config (default: skip if exists)

syntaur create-mission <title>
  --slug <slug>           # Override auto-generated slug
  --dir <path>            # Override default mission directory

syntaur create-assignment <title>

  --one-off               # Auto-wrap in a new single-assignment mission
  --slug <slug>           # Override auto-generated slug
  --priority <level>      # low|medium|high|critical (default: medium)
  --depends-on <slugs>    # Comma-separated dependency slugs
```

## Key Architecture Decisions Needed

1. **Commander.js vs yargs** — Commander.js is recommended. It's the industry standard, has first-class TypeScript support, and uses a clean fluent API for subcommand definition.

2. **Build tool** — tsup is recommended. It produces clean bundled output, handles TypeScript natively, and requires minimal configuration.

3. **ESM vs CJS** — ESM primary output. Node 20+ has full ESM support. The bin shim can use `#!/usr/bin/env node` with `"type": "module"` in package.json.

4. **YAML serialization** — Need a library to generate YAML frontmatter. Options: `yaml` (npm package) for proper YAML serialization, or simple string templates since the frontmatter is well-defined. String templates are simpler and avoid a dependency; `yaml` package is safer for escaping edge cases in user-provided titles.

5. **UUID generation** — `crypto.randomUUID()` is built into Node.js 20+. No external dependency needed.

6. **Interactive prompts vs flags-only** — Start with flags-only for v1 (simpler, scriptable). Interactive prompts can be added later.

7. **Slug generation** — Convert title to slug: lowercase, replace spaces/special chars with hyphens, strip non-alphanumeric, collapse multiple hyphens. Validate: must be lowercase, hyphen-separated, no leading/trailing hyphens, no consecutive hyphens.
